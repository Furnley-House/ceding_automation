New files / changes:

## 4 June 2026 — Data integrity & schema cleanup

### Checklist–Case mirror

When AI extraction (or test-fill / manual edit) updates `provider_name`, `plan_number`, or `start_date` in the checklist, those values are now propagated to the corresponding Case columns so the header, cases list, and dashboard reflect the latest extracted value.

- `backend/src/services/caseFieldMirror.ts` — new `mirrorChecklistToCase` helper
  - `provider_name` → upserts Provider record, sets `Case.providerId`
  - `plan_number` → `Case.policyRef`
  - `start_date` → `Case.planStartDate` (parses DD/MM/YYYY or ISO)
- Called from: `applyFieldExtraction`, PATCH `/checklist/:fieldId`, POST `/checklist/seed`, POST `/checklist/fill-test-data`
- Idempotent — skips writes if the column is already in sync

### Checklist source-of-truth reconciliation

Admin Panel showed 51 ISA / 69 Pension fields while Extract & Fill Gaps showed 33 ISA. Root cause: the DB had 63 orphan templates from older canonical versions that the seed never deactivated against the current canonical (`Ceding_Checklist_-_Blank.xlsx` v1.1.0, confirmed with Reva 2026-05-29).

- Built `backend/src/scripts/diffTemplates.ts` diagnostic
- Re-ran `seed.ts` → deactivated 63 orphans (18 ISA + 12 GIA + 33 Pension)
- Both Admin Panel and Extract view now show the same canonical truth

### Fund Details sheet in XLSX

Added 4th sheet to the export workbook (used by both the local download and the WorkDrive upload):

```
Summary | Checklist | Fund Details (NEW) | Audit Trail
```

Fund Details columns: Fund Name · ISIN · Units · Price · Value · Charge · With-profits · Status · Confidence · Source page · Last updated. Includes a TOTAL row at the bottom.

### DB duplicate audit — `acceptedSigType`, `planSubType`, CallScript snapshots, duplicate notifications

Built `backend/src/scripts/auditDuplicates.ts` (later deleted post-cleanup). Findings:

| Item | Status | Action |
|---|---|---|
| `Provider.acceptedSigType` (free string) | Pure duplicate of `Provider.loaFormat` (enum) — 3 rows had matching values | **Dropped** |
| `Case.planSubType` + `PlanSubType` enum | Never populated (0 of 3 cases) | **Dropped** |
| `CallScript.providerPhone` + `providerDept` | Stale snapshots; same data already inside `scriptContent` JSON | **Dropped** |
| `CaseStatus.IN_REVIEW` vs `STAGE_9_ADVISER_REVIEW` | Semantically overlapping but high-risk to migrate mid-test | Deferred; mitigated by `STATUS_TO_STAGE` mapping |
| **Duplicate notifications** — 4 rows for same `(userId, caseId, title)` | Stepper + status route each fired separate notifications | **Deleted live**; added `maybeCreateNotification` with 5-minute dedup window |

### Schema migration

New migration: `prisma/migrations/20260604120000_drop_redundant_columns/migration.sql`

```sql
ALTER TABLE "providers"    DROP COLUMN IF EXISTS "acceptedSigType";
ALTER TABLE "cases"        DROP COLUMN IF EXISTS "planSubType";
ALTER TABLE "call_scripts" DROP COLUMN IF EXISTS "providerPhone";
ALTER TABLE "call_scripts" DROP COLUMN IF EXISTS "providerDept";
DROP TYPE IF EXISTS "PlanSubType";
```

Applied non-interactively via `prisma migrate deploy` (the destructive-warning prompts in `migrate dev` block non-interactive environments).

Touching code updated: `prisma/schema.prisma`, `prisma/seed.ts`, `routes/providers.ts`, `routes/cases.ts`, `routes/checklist.ts`, `routes/calls.ts`, `services/zohoCrm.ts`. Deleted obsolete `src/scripts/auditDuplicates.ts`.

### Cached Zoho IDs on Case (earlier 4 Jun migration)

`prisma/migrations/20260603182332_cache_zoho_ids_on_case/migration.sql` — added:
- `zohoOwnerId String?`
- `zohoClientOwnerIds String[]`

## Known follow-ups (deliberately deferred)

1. **`CaseStatus` enum consolidation** — `IN_REVIEW`, `STAGE_9_ADVISER_REVIEW`, `APPROVED` overlap. Currently mapped via `STATUS_TO_STAGE`. Would need a careful backfill if formally collapsed.
2. **`Client_Owners` field on Plans PATCH** — disabled. Field is Multi-Select Lookup but the IDs we have don't resolve cleanly. Workaround: CRM workflow can copy from Contact.
3. **OAuth scope refresh** — re-authorisation needed in dev for the `ZohoCRM.modules.ALL` + `WorkDrive.files.ALL` scopes added during this work. The refresh-token in `.env` may need updating after the consent page is re-run.
4. **Two copies of canonical JSON** — `backend/prisma/canonical/checklist-fields-v1.json` and `frontend/src/lib/canonical/checklist-fields-v1.json`. Build-time sync-check would prevent drift.
5. **Admin Panel writes are ephemeral** — the seed re-applies canonical, so any new fields added via the UI are deactivated on next seed run. Either lock Admin to read-only or stop running the seed in environments where Admin edits should persist.
6. **Stage 4 / Admin Panel field counts** — Admin shows 34 (incl. fund_lines table row), Extract shows 33 (frontend filters table type into the dedicated FundDetailsTable). Cosmetic off-by-one.
